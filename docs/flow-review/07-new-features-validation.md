### 사용자 결정 4가지 — RESOLVED ✅ (상세 §4)
1. **F3 출금동의 모델** → **시간별 위약금 + 전액출금** (무임승차·교착·사비손실 동시 소멸, §6 개정 필요)
2. **F3 OCR 수정 거버넌스** → **수정+원본노출만** (목업 허용, 실서비스 전 보완 필요)
3. **F1b 출금시각 의미** → **만남시각 이후 지연만** (`출금시각 ≥ lockAt`, 동의게이트 유지)
4. **F2 번개 머니 모델** → **기존 개인 번개 + 5,000원 예약금, 에스크로 경유** (게스트→관리자 직접송금 차단, Step1 KYC)

> **결과**: 원 명세("무응답=거절+부분출금")가 깨던 정산경제를 "시간별 위약금+전액출금"이 해소. CONSENT 데드락(F3-3·A5) 원천 소멸. /spec 진입 준비 완료.

## 1. 기능별 모순·병목·엣지케이스 (심각도순)

### F3 — 5분 비동기 출금동의 (최고 위험)

이 기능은 문서화된 **전원동의 모델**(`allAgreed=true → 전액 출금`, 03 §3)을 조용히 **부분동의/부분출금**으로 바꾼다. 아래 대부분이 그 교체에서 파생.

| ID | 발견 | 시나리오 | 위협 불변식 | 심각도 |
|---|---|---|---|---|
| **F3-1** | 무임승차 + 총무 사비손실 (장부는 맞음) | 8명×4만=32만 예치. 5명 동의(20만), 3명 무응답→자동거절+환불(12만). 8명 다 먹음, 식대 28만. 총무는 20만만 출금→**8만 사비**, 무응답 3명은 공짜로 먹고 환불받음. REFUND_CANCEL이 장부는 맞춰서 **모델이 망가져도 잔액=분개합은 성립** | §4-6 차수원자성·§1 투명정산 (§4-4 아님) | **CRITICAL** |
| **F3-2** | 타이머×수동 출금 경합 → 진짜 이중지급 | 5분 자동거절 잡과 막판 수동출금이 동시 커밋. 직렬화 락+멱등키 없으면 32만 전액출금 + 12만 환불이 둘 다 커밋 → **에스크로 음수, 12만 이중지급** | §4-4 잔액=분개합·§4-5 멱등 | **CRITICAL** |
| **F3-4** | 인라인 알림액션 = 세션 없는 금전 승인 | 잠금화면 푸시의 [동의] 탭이 32만원 방출을 인가. 토큰이 APNs/FCM(3자) 경유·기기잠금 없이 발동·단발성 아니면 재사용. 분실/공유 폰의 탭이 자금방출 인가 | §4-7 인증게이트·§4-8 AuditLog | **CRITICAL** |

| **F3-8** | 부원 이의제기(dispute) 경로 부재 | OCR이 수정가능해지면 강제노출 사진은 장식. 부원은 위조 의심 숫자를 **볼 수는 있으나 정산을 막을 장치가 없음** | §1·§4-8 | **CRITICAL** |
| **A5** | 5분 기본거절 + cancel-refund 병존 = 거의 모든 모임 자동취소 | 전원동의 유지 상태에서 8명이 5분 내 전원 응답은 비현실 → **출금요청 5분 내 거의 항상 자동 취소·환불** → 총무가 실지출을 영영 정산 못함 | 모델 실행가능성·§1 | **CRITICAL** |
| **F3-3** | 선택적 환불의 합법 상태 없음 | LOCKED는 취소 차단(`409 LOCKED`). REFUNDING 탈출은 **전원** 취소→DONE용이지 "일부 환불+나머지 출금"용이 아님 | Escrow 상태머신 불변식 | HIGH |

| **F3-9** | "우선 출금 권한"은 원장에 없는 원시명령 | 동의자 지분만 부분출금 = `출금액 ≠ 차수 예치합`. 계약의 `Payout`은 단일 전액. 부분출금된 에스크로 위 균등정산(DONE→0)의 수학 미정의 | §4-6 차수분개·DONE=0 불변식 | HIGH |

| **C-k** | 정산피드 광고 제거는 04 §3의 "출금직후 광고대기 3~5초" 제거 | 무해한 흐름변경, 불변식 영향 없음 (사용자 요청대로 제거 OK) | — | LOW(승인) |

## 2. 협상 불가 불변식 (사용자 결정 아님 — 구현 시 무조건 준수)

아래는 "옵션"이 아니라 **올바름 요구사항**. 어떤 결정을 고르든 지켜야 함:

1. **5분 타이머는 서버 권위**: 클라 카운트다운 표시만, 만료판정은 `요청시각+300s` vs **서버 수신시각** (F3-5).
2. **타이머 만료 처리는 멱등 + 모임단위 직렬화 락**: 타이머 발화와 수동 출금이 정확히 한 지점에서 수렴 (F3-2, 04 §5).
3. **인라인 알림 액션은 단발성·수신자 한정·짧은 만료 토큰** + 실제 인증 주체로 해소 (F3-4).
4. **게스트 초기 신뢰값은 서버 발급** — 클라 계산·하드코딩 금지, 클라는 렌더만 (F2-3, §11).
5. **게스트→회원 전환은 동일 JoinApplication/Membership 머신** 종결 — 정원가드 `409 CLUB_FULL` 통과 (C-f).
6. **번개 정산은 가입승인과 직교** — 머니는 사회적 결정과 분리 (F2-6).
7. **번개 미생성/방치 종료상태 명명** — `POLL_TIE·POLL_EXPIRED·TRIAL_NOSHOW·TRIAL_APPROVAL_TIMEOUT·TRIAL_WITHDRAWN` (F2-5).
8. **overdue/패널티 스케줄러는 `lockAt` 기준** — 총무의 자발적 출금요청과 독립 (F3-6, F1b-3).
9. **지도 링크는 `https:`만 + 검증**, 자유텍스트 place와 분리 (F1c-1).
10. **정산피드 광고 완전 제거** — 사용자 요청 승인, 04 §3 광고대기 단계도 삭제 (C-k).

## 3. "지금 만들 화면" vs "백엔드 불변식(시뮬레이션만)" 분리

> /spec은 **화면**을 만든다. 아래 좌측만 목업으로 구현하고, 우측은 목업이 *흉내만* 낸다(실제 머니수학 구현 금지).

### F1
- **만들 화면**: 모임생성 폼에 ①게스트 토글 ②출금시각 타임피커 UI ③장소 자유텍스트+지도링크 필드.
- **백엔드(흉내만)**: 출금시각의 *의미*(목업은 락/동의에서 출금을 실제 분리하지 말 것 — 결정3 전까지 만남시각 별칭/표시값으로). 락 스케줄러는 서버측.

### F2
- **만들 화면**: Step1 게스트 신청 폼 · Step2 관리자 뷰(신청자 수 + 날짜투표 컴포넌트) · Step3 가입승인 버튼 · 3-step 네비 흐름.
- **백엔드(흉내만)**: 번개 자동생성(서버가 모임+에스크로+소유권 생성) · KYC 검증결과(목업은 서버 `verified` 플래그로 게이팅, KYC 자체 미구현) · 신뢰검증 결과는 **서버 배지**(클라 계산 금지) · JoinApplication→Membership 전환+정원가드 · 번개 에스크로 멱등/원장.

### F3
- **만들 화면**: 광고없는 결제/정산 피드 · 5분 동의 타이머 UI+카운트다운 · 동의투표 화면 · **OCR 수정 입력폼 + 하단 원본 영수증 강제노출** · 인라인 알림 [동의/거절] 액션 UI.
- **백엔드(흉내만)**: 5분 만료→기본거절 전이(서버 타이머; 목업은 "5분 경과→거절 처리됨" 표시만) · 부분동의 출금수학·"우선출금권한" 지분계산(결정1 전까지 원장타입 미정) · `잔액=Σ원장`·OCR→원장 바인딩·`422 SETTLEMENT_INVALID` 검증 · settlementDeadline 타이머. **목업은 상태만 시뮬레이션, 머니수학 미구현.**

## 4. RESOLVED DECISIONS — 확정 (/spec 입력) ✅

> advisor + 양 서브에이전트가 수렴한 갈림길을 사용자가 확정. 아래가 /spec 화면 명세의 전제.

### 결정 1 — F3 동의/위약금 모델 ✅ **[시간별 위약금 + 전액출금]**
원 명세("무응답=거절 + 동의자 지분만 부분출금")는 무임승차·총무 사비손실·자동취소(A5)로 정산경제를 깨므로 **폐기**. 확정 모델:
1. **신청 기간 = 예치(DEPOSIT)**: 신청과 동시에 차수 비용 예치.
2. **신청마감 → 만남시각 = 시간별 취소 위약금**: 늦게 취소할수록 차감↑(예시 티어: 마감~24h전 0% · 24h~6h 50% · 6h~만남 100%). 위약금은 **에스크로 잔류 → 공동비용/참석자 환급에 충당**(플랫폼·총무 귀속 **0** → §11 준수). 차수별 분개(REFUND_CANCEL은 위약금 차감분만).
3. **무응답 = 거절 = 수수료 부담 노쇼**: 환불 없음, 예치 유지 → **공짜 식사 불가, F3-1 무임승차 해소**.
4. **오프라인 만남 후 총무 "출금 동의받기" → 5분 알림**: **동의 = 투명성 확인, 거절 = 이의 기록**(출금을 막지 않음).
5. **전액 출금**: PAYOUT = 에스크로 전액(무응답자 몫도 위약금 부담으로 포함) → **총무 사비손실 0, 부분출금 원장타입 불필요**.
- **부수효과**: 출금이 `allAgreed`에 안 매달려 **CONSENT 데드락(F3-3)·5분 자동취소(A5) 원천 소멸**. 신규 "우선출금권한"(F3-9) 불필요.
- ⚠ **CLAUDE.md §6 개정 필요**: "만남시각 전 취소·환불 자유" → "신청마감 후 시간별 취소 위약금". *office-hours 범위 밖 — /spec 또는 별도 커밋에서 §6 1줄 개정 권장.*

### 결정 2 — F3 OCR 거버넌스 ✅ **[수정+원본노출만]**
OCR 추출값 사용자 수정 가능 + 하단 원본 영수증 강제노출. 추가 거버넌스 없음.
- ⚠ **잔여 리스크(목업 허용, 실서비스 전 보완 필요)**: F3-7 상향 착복(실 5만→32만 입력→27만 착복) 강제력 0. 실서비스 전환 시 **수정 AuditLog + 부원 이의제기 + 정수·`≤출금액` 검증** 추가 권장.

### 결정 3 — F1b 출금 가능시각 ✅ **[만남시각 이후 지연만]**
타임피커는 `출금시각 ≥ lockAt`만 허용(만남 전 출금 차단). 출금은 여전히 만남시각 락 + 총무 동의게이트(결정1). "총무가 출금 시작 시점을 뒤로 미룸" 의도만 충족 → **F1b-1·F1b-2·A1 위반 제거**.

### 결정 4 — F2 번개 머니 모델 ✅ **[기존 개인 번개 + 5,000원 예약금, 에스크로 경유]**
번개 = 기존 개인 번개 동일 구조, hostId=관리자, **차수당 5,000원 노쇼 예약금**. §11 준수 위해:
- 게스트 예치 → **모임 키 시스템 에스크로** → (만남시각 락 + 동의) → 관리자 **본인 계좌 PAYOUT 출금**. **게스트→관리자 직접 송금 차단**(에스크로 경유 필수, F2-1 해소).
- **Step1에서 KYC 요구**(5,000원 예치는 verified 게이트, B/F2-4 닭-달걀 해소).
- 가입 승인(Step3)은 **정산과 직교**(F2-6) + **동일 JoinApplication 머신**(정원가드 `409 CLUB_FULL`, C-f).

### §2 불변식 적용 갱신 (결정 반영)
- 결정1 전액출금으로 **#2(타이머×수동 경합)는 여전히 필수**(직렬화 락·멱등) — 단 부분출금 수학은 불필요.
- **#3(인라인 알림 토큰)·#1(서버 5분 시계)·#4(게스트 신뢰 서버발급)·#7(번개 종료상태)·#8(overdue=lockAt 기준)** 전부 유효.
- 결정2로 OCR 거버넌스 보완장치는 **목업에선 생략**(실서비스 TODO).
