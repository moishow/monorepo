개요
01 · Document Overview
문서 개요 · 용어

본 PRD는 지금까지 확정된 모이쇼의 디자인·기능을 빠짐없이 집약해 백엔드 설계(데이터 모델·상태 전이·API)의 기준을 제공한다. 화면 스펙(와이어프레임), UI 킷(앱·어드민 목업), 디자인 시스템을 단일 소스로 통합했다.

용어정의
포인트 원장(LedgerEntry)모든 포인트 이동을 append-only로 분개. 유저/모임 잔액 = 분개 합으로 재계산 가능(가상 장부).
포인트 에스크로모임/차수 단위로 예치 포인트를 보관. 만남시각 락 → 전원 동의 → 총무 출금 → 반납·정산. 카카오페이는 충전·현금화 레일로만 사용.
안심 예산안모임 차수의 항목별 사용처를 사전 공개하는 신뢰 장치. 모임 상세(AP-05)의 핵심.
충전 대사(Reconcile)카카오페이 충전(webhook) 내역과 지갑 포인트 적립을 대조·점검(누락/불일치 보정). 차수 예치는 보유 포인트 차감.
총무 vs 운영자총무=모임 단위 운영(앱 내장 도구). 운영자=본사 플랫폼 운영(웹 어드민). 권한이 의도적으로 분리됨.
⚠ 머니플로우 정본 변경 — 아래 결제 서술은 초기 설계(폐기) · 이 PRD의 결제·정산 모델(에스크로 통합계좌 + 카카오페이 이체 + 운영자 입금 대사(AD-10) + 지급 승인, Campaign·Contribution·Payout)은 초기 설계다. 현재 정본은 앱 포인트 기반 기간 제한형 에스크로 — 예치 → 락 → 전원 동의(CONSENT) → 총무 출금 → 증빙·반납 → 자동 정산 — 이며 openapi.yaml §7·§8 + 05_포인트에스크로정산모델 이 정본이다(거래 단위=차수, 회원 간 송금 없음, 포인트는 모임 정산 목적 외 사용 불가, 충전·출금 수수료 0%).
02 · Product
제품 개요 · 콘셉트

모이쇼는 동아리·소모임의 회비·펀딩을 모으고 → 예산을 짜고 → 투명하게 정산하는 서비스다. "신뢰감을 주는 활기찬 핀테크"를 지향한다.

신뢰 (Trust)

계정계 수준의 정확함. 금액·인원·달성률 등 숫자가 화면의 주인공. 안심 예산안·감사 로그·에스크로로 투명성 확보.

활기 (Vibrant)

모임의 즐거움. 쇼/Showts 소셜, 이벤트 배너, 위트 있는 독촉 보이스로 가벼운 진입.

친근 (Friendly) · "정중하지만 위트 있는 총무 친구"

존댓말·권유형 카피, 메시지당 이모지 1개. 돈이 오가므로 시스템 메시지는 명확하되, 독촉·미입금 안내는 유저가 상처받지 않게 처리.

05 · Core Domain
핵심 도메인 · 머니플로우

자금은 항상 에스크로 통합계좌를 거친다. 유저/모임의 잔액은 실제 출금이 아니라 가상 장부의 분개(LedgerEntry)로 표현된다.

→

→
AD-10입금 대사
→
자동/수동확인
→
AP-10장부 반영

이름+금액 일치 시 자동 매칭(자동매칭률 KPI). 불일치(가족 명의·부족·초과)·미입금은 운영자 수동 확인. 확인된 납부만 raisedAmount에 집계.

정산 → 지급 승인 → 환급

AP-06실지출 입력
→
자동계산차액·1인당 환급
→
AP-06정산 마감
→

→
에스크로환급/지급 이체
→
AP-10장부 반영

총무가 마감하면 Payout(대기) 생성 → 운영자 승인 시 이체 실행 + LedgerEntry(지급/환급) 기록. 실패 건은 AD-05에서 재처리.

거래 타입 (LedgerEntry.type)

타입부호설명
펀딩 / 회비+유저 → 에스크로 입금(납부).
지급−에스크로 → 총무 정산 지급.
환급−에스크로 → 유저 환급(차액·취소).
수수료−플랫폼 수수료 자동 차감(초기 0%).
원장은 append-only(불변)로 설계해 감사 추적을 보장한다. 잔액은 항상 LedgerEntry 합으로 재계산 가능해야 한다(이중지급·정합성 검증).

07 · State Machines
상태 머신

핵심 리소스의 상태 전이. 전이 이벤트는 AuditLog로 기록한다.

Campaign (캠페인)

검토→진행중→마감임박→정산대기→완료→정지

개설 심사 통과 → 진행중. 마감 D-3 이내 자동 마감임박. 100% 또는 마감 → 정산대기 → 정산 완료. 이상 시 정지(신규 납부 차단).

Deposit (입금 대사)

이체완료표시→자동매칭→수동대기→확인완료→미입금

유저 "이체 완료" → 카카오페이 내역 대조. 이름+금액 일치 = 자동매칭 → 장부 반영. 불일치(이름/금액)·미입금 = 수동대기 → 운영자 확인.

Payout (지급 승인)

대기→승인→반려→실패→완료

총무 요청(대기) → 운영자 승인 → 이체 실행(완료). 반려 시 사유 통보. 이체 실패 → 재처리(AD-05).

KYC (본인인증)

미인증→대기→인증→보류→실패

가입 직후 미인증 → 인증 시도. 통과=인증(지갑 개설). 미흡=보류. 5회 실패 시 30분 잠금.

Group (모임 심사)

심사→활성→정지

신규 개설 → 심사. 승인=활성, 반려/위반=정지. 정지 시 캠페인·거래 동결.

Post 모더레이션

정상→신고→검토→숨김→예약

신고 누적 → 검토 큐. 운영자 유지=정상, 숨김 → 작성자 통보. 배너는 예약.

Showts 검수

검수대기→신고→승인→반려

업로드 → 검수대기. 승인 시 피드 노출. 반려 시 사유 통보. 신고 누적 별도 큐.

CSTicket (고객센터)

미처리→처리중→완료

문의 유입(미처리) → 첫 답장 시 처리중 자동 전환 → 해결 시 완료. 환불 재처리 연동.

PushCampaign (푸시)

작성→예약→발송완료

타겟·메시지 구성 → 즉시 발송완료 또는 예약. 발송 후 열람률 집계.
