개요
How to read
이 문서를 읽는 법

각 엔티티 카드는 설명 · 출처 화면(우상단 모노폰트) · 필드 표(필드/타입/값·예시) · 관계로 구성된다. 필드명·타입·값은 목업 코드에서 그대로 추출했으며, 백엔드 스키마 확정 시 정규화·인덱스·제약을 보강한다. 핵심 표시는 PRD 대비 새로 드러났거나 비중이 큰 도메인이다.

⚠ PRD와의 관계 · 앞서 만든 PRD는 와이어프레임 스펙 기준이라 매너온도·신뢰등급, 팔로우, 차수(Round), 번개 노쇼 예약금, 모임통장/오픈뱅킹, 장부 포인트 환불, 폼빌더 등이 빠져 있었다. 본 정의서는 실제 목업 구현을 기준으로 이를 모두 포함한다.

머니/결제 모델 · 오픈 이슈

목업에서 드러난 결제·정산 흐름은 3가지로 갈린다. 백엔드 결제 도메인 설계의 출발점이다.

결제·정산 모델

동아리 정기 모임차수별 회비 → 카카오페이 선송금(합산) 또는 모임통장 자동 수취 → 모임 후 정산 → 잔액 1인당 환급(계좌 or 장부 포인트)
개인 번개(flash)차수당 노쇼 예약금 5,000원만 선송금 → 현장 후정산(잔액) → 노쇼 시 예약금 몰수, 고정비 환불 불가
동아리 장부정산 잔액이 모임통장/장부에 적립 → 다음 모임 펀딩 시 자동 차감(페이백)

백엔드 확정 필요 — 오픈 이슈

결제 연동 불일치앱 설정엔 “오픈뱅킹 연동”, 어드민 입금확인엔 “은행/PG 미연동·수동 대사”. 어느 쪽이 1차 출시 기준인지 확정 필요.
신뢰점수 산식score·매너온도(temp)·등급 컷의 계산식/갱신 트리거(정산·출석·칭찬·노쇼)를 백엔드에서 정의해야 함.
차수(Round) 결제 단위신청·정산·환불이 모임 단위인지 차수 단위인지 — 데이터상 차수별 인원/비용이 독립이므로 차수 단위 권장.
환불 방식 2종계좌 환불 vs 장부 포인트 적립(다음 모임 자동 차감) — 회계상 분개·잔액 처리 분리 필요.
역할 4단계부원/운영진/총무/회장 — 권한 매트릭스(모임개설·장부마감·가입승인·지급요청) 확정 필요.
권장 · 결제·정산은 차수(Round) 단위를 1차 거래 단위로 두고, 모임/동아리는 그 집계로 설계하면 목업의 다중 차수·번개 예약금·부분 환불을 모두 수용한다.
