결제 시퀀스 · 상태
Payment Sequence
카카오페이 결제·환불 시퀀스

approve(동기 복귀)와 webhook(비동기)이 같은 결제를 멱등하게 합류시켜 이중 확정을 막는다.

1. App → ServerPOST /meetings/{id}/attendances {roundIds} → att(pending_payment), amount
2. App → ServerPOST /payments/kakao/ready (Idempotency-Key) → tid·redirect/appScheme
3. App → KakaoPay앱 스킴/리다이렉트로 결제창 진입 → 사용자 승인
4. KakaoPay → Apppg_token 반환(redirect 복귀)
5. App → ServerPOST /payments/kakao/approve {tid, pgToken}
6. KakaoPay → ServerPOST /payments/kakao/webhook {SUCCESS_PAYMENT} (비동기·서명)
7. Serverapprove+webhook 멱등 합류 → payment=completed, 차수 att=paid, 장부 분개, 알림
실패·복구 · webhook 미수신 시 ready 시 발급한 tid로 주문 조회 폴링 폴백. approve 실패+webhook 성공이면 webhook 기준으로 확정. 환불도 동일 멱등 규칙.

상태 전이 요약

Attendancepending_payment → paid → (cancelled / noshow)
Paymentready → approved/completed → (failed / refunded)
Settlementopen(작성중) → closed(마감) ＋ refunds: processing → completed
Refundrequested → processing → completed / failed
차수 단위 원칙 · 한 모임에 여러 차수를 신청하면 결제는 1건(합산)이지만 차수별 Attendance·LedgerEntry로 분개되어, 일부 차수 취소·노쇼·부분 환불이 독립 처리된다.
